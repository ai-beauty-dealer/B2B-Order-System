# 出荷PDF取込 セットアップ・運用手順

## できること

倉庫入出庫一覧表PDFを読み込み、PDFの日付から通常シートと直送シートを自動判定して照合する。

対象日はPDF左上の印刷日ではなく、商品明細の「日付」列から取得する。過去日のPDFを後日印刷しても、明細日付と同じ過去シートを選ぶ。現在開いているシートは対象判定に使わない。

- 発注サイトの既存注文
  - `入力数 - 出荷数` が1以上の行だけ、I列「欠品数」へ反映
  - 欠品がない行のI列は空欄
- 発注サイト外の商品
  - PDF区分が `サロン直` で、`YYYY-MM-DD直送` シートが存在する場合は直送シート
  - `サロン直`でも直送シートがない場合は通常シートへまとめる
  - それ以外は `YYYY-MM-DD` シート
  - 対象シートの最下部へ、B列「商品コード」・C列「出荷数」・D列「商品名」だけ追加
  - E列「得意先名」は手入力
  - F列以降は変更しない
  - 通常シートもない場合は、プレビューに「新規作成」と表示し、反映時に通常シートを自動作成する
- 独立した欠品管理ファイル
  - 発注スプレッドシートとは別のGoogleスプレッドシート「欠品管理」を作成する
  - `欠品リスト`シートのA列「商品コード」・B列「商品名」・C列「個数」・D列「サロン名」・E列「欠品日時」へ保存する
  - 既存の4列形式は自動でB列に「商品名」を追加し、再同期時に既存行の商品名を補完する
  - 欠品日時にはPDFの対象日を保存する
  - 同じ対象日・商品・個数・サロンの重複追加を防ぐ
- 除外コード
  - `9999001`
  - `9999991`
  - 完全一致だけを除外する。`9997`から始まる実商品は除外しない

同じPDFを再度反映しないよう、非表示シート `_出荷PDF取込ログ` にPDFハッシュと結果を保存する。

## B2B発注Apps Scriptへ追加するファイル

既存の `code.gs` と同じApps Scriptプロジェクトへ、以下の3ファイルを追加する。

| Apps Script上の種類・名前 | ローカル正本 |
|---|---|
| スクリプト `shipping_import` | `backend/shipping_import.gs` |
| HTML `shipping_import_ui` | `backend/shipping_import_ui.html` |
| HTML `shipping_pdf_parser` | `backend/shipping_pdf_parser.html` |

### 追加手順

1. B2B発注サイトで使っているApps Scriptプロジェクトを開く。
2. 左側の「ファイル」横の `+` から「スクリプト」を選び、`shipping_import` を作る。
3. `backend/shipping_import.gs` の全内容を貼る。
4. 「HTML」を選び、`shipping_import_ui` を作る。
5. `backend/shipping_import_ui.html` の全内容を貼る。
6. もう1つHTML `shipping_pdf_parser` を作る。
7. `backend/shipping_pdf_parser.html` の全内容を貼る。
8. 3ファイルを保存する。
9. 発注スプレッドシートを再読み込みする。
10. 上部メニューに「出荷PDF」が出れば追加完了。
11. 「出荷PDF」→「欠品管理ファイルを作成・確認」を1回実行し、権限を許可する。
12. Googleドライブに独立ファイル「欠品管理」が作成される。

メニューが出ない場合は、Apps Scriptの関数一覧から `installShippingImportMenu` を1回実行してスプレッドシートを再読み込みする。

この機能はスプレッドシート内から動くため、Webアプリの再デプロイは不要。既存の発注サイトAPIには変更を加えない。

## 日々の使い方

1. 発注スプレッドシートを開く。
2. 「出荷PDF」→「PDFを取り込む」を押す。
3. 当日の倉庫入出庫一覧表PDFを選ぶ。
4. 「PDFを解析する」を押す。
5. 表示された次の数字を確認する。
   - PDF商品明細
   - 発注サイト照合
   - 欠品合計数
   - サイト外追加行
6. 問題なければ「この内容で反映する」を押す。
7. 通常・直送シートの最下部に追加された発注サイト外商品のE列へ、得意先名を入力する。
8. 納品書集約ツールの `executeAggregation` を実行する。

すでに発注シートI列へ反映済みの欠品は、「出荷PDF」→「既存の欠品をリストへ反映」を1回実行すると独立ファイルへ同期できる。

## 納品書集約ツールの更新

納品書側のApps Scriptは、ローカル正本 `99_Sbox/00_納品書集約ツール_完成版.js` へ差し替える。

更新後は、発注シートのI列「欠品数」を集計し、納品書シートへ次の形で出力する。

| 列 | 内容 |
|---|---|
| A | 得意先名 |
| B | 集約コード |
| C | 集約後商品名 |
| D | 合計数量 |
| E | 欠品（数量） |
| F | ステータス（納品書入力後に手動で `完了`） |
| G | 備考（既存列、集約時に変更しない） |

発注スプレッドシート側のJ列は「移動先シート日付」。納品書側のJ列ではない。

F列へ欠品判定は自動入力しない。納品書変換を再実行した場合も、同じ得意先名・集約コードの既存ステータスを引き継ぐため、入力済みの `完了` は消えない。

### 別日の納品書へ移動する

1. 移動元の発注スプレッドシートJ列「移動先シート日付」へ `YYYY-MM-DD` 形式で入力する。
2. `executeAggregation` を実行し、移動元の日付を入力する。
3. 移動先の日付シートがなければ自動作成し、対象商品を移動する。
4. 移動情報は非表示シート `_納品日移動管理` へ保存する。

移動後に移動元を再同期しても商品は元へ戻らない。移動先を発注サイトから再同期しても、移動した数量は消えず、同じ得意先・商品コードの数量と合算する。
すでに移動管理へ登録済みの移動先も毎回再構築するため、移動先シートだけ作成され、商品行が空になった場合でも移動元日の再実行で復旧する。

2026-07-01の発注スプレッドシートの例：

- 45行目「アイビル タフタイマー」のJ45へ `2026-07-06`
- 46行目「サーモローラー」のJ46へ `2026-07-02`
- その後 `executeAggregation` で `2026-07-01` を指定

### F列ステータスの運用

- 発注サイトの通常・直送シート：入力処理後に手動で `完了`
- 納品書シート：納品書入力後に手動で `完了`
- PDF取込処理は、どちらのF列も変更しない

`CONFIG.ORDER_SPREADSHEET_ID` はApps Script側で現在使っている実際のIDを残す。ローカル正本のプレースホルダーで上書きしない。

## 照合方法

発注サイトの注文行はA列のタイムスタンプで注文単位にまとめ、PDF側の送信日時・商品コード・入力数を使って照合する。

- PDF区分 `サロン直` は直送シートだけと照合
- それ以外は通常シートだけと照合
- 同じ送信時刻に複数注文がある場合は、商品コードと数量の組み合わせが最も一致する注文を選ぶ
- 伝票や送信時刻のまとまりが一致しない明細は、最後に対象シート全体を商品コード×個数で再照合する
- PDFにしかない商品は発注サイト外として扱う
- 商品名中の容量 `85`、`100`、`2000` などは、PDF上の列位置で分離するため数量として誤読しない
- 左上の印刷日と明細日付が違う場合は、明細日付を優先する

- 伝票グループ数: 5
- PDF明細数: 74
- `9999001`・`9999991`除外後: 70

- 印刷日を2026-07-01のまま、明細日付だけ2026-06-30にしたテストで対象日`2026-06-30`を取得
- 模擬欠品データでI列へ欠品数 `1` を反映
- 発注サイト外行はB・C・D列だけ書き、A・E・F・G・H・I列を空欄のままにすることを確認

## 注意

- 反映前に必ずプレビューの件数を確認する。
- PDFレイアウトが変わった場合は、解析結果の明細数が不自然になる可能性がある。その場合は反映せず、PDFを確認する。
- すでに得意先名を入力した発注サイト外行は、A列が空欄でもB・C・Dが一致すれば二重追加しない。
